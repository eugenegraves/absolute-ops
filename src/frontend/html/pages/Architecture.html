Architecture

One server. The right UI per route.

AbsoluteOps runs on a single Bun process with Elysia routing. Each route picks the rendering strategy that matches its job — static for content, HTMX for thin server-driven flows, React/Svelte/Vue/Angular for app-like surfaces, and the AbsoluteJS aiChat plugin for streamed AI. The Framework Inspector overlay on each page makes the choice visible.

Stack at a glance

No microservices, no glue. AbsoluteJS resolves the manifest at boot, mounts each handler, and serves both pre-rendered HTML and SSR responses from the same Elysia app.

Server

  • Bun — runtime + bundler + test runner
  • Elysia — typed HTTP routing
  • AbsoluteJS — pages, islands, AI, pre-render, ISR
  • elysia-scoped-state — per-session UI state

Client

  • HTMX for server-driven flows
  • React 19 for the operator workspace
  • Svelte 5 for the executive dashboard
  • Vue 3 for the AI briefing
  • Angular 21 SSR for the enterprise admin

Routes & render strategies

Static generation, server-rendered HTMX fragments, SSR + hydration, and WebSocket streams — all picked per route, not per app.

Why this matters

Most teams pick one frontend stack and contort every surface to fit. AbsoluteJS lets you keep the right tool for the right job — without a second deploy target.

Public surfaces stay light

Marketing and industry pages are pre-rendered HTML. No framework runtime ships unless a page actually needs one.

Customer flows stay simple

HTMX fragments mean the customer-facing tracker and the field-tech app load fast even on weak networks and old phones.

App surfaces stay rich

The operator workspace runs React because dispatchers need real UI affordances. The dashboard runs Svelte because it ticks every five seconds.
